Each layer is a “holy shit” escalation. Non-technical people see the compounding. Technical people see the architecture. Both are floored for different reasons.

Here’s the answer on GWS CLI scopes.

Option C is what you want: a separate OAuth client with restricted scopes.

  1. Go to the GCP project
  2. Create a new OAuth 2.0 Desktop Client — name it “Workshop Attendees — Read Only”
  3. On the OAuth consent screen, enable ONLY these scopes: spreadsheets.readonly, drive.readonly, gmail.compose (allows drafts, blocks send and delete)
  4. Give attendees the client ID + client secret
  5. They run gws auth login — Google only grants tokens with those 3 scopes

This is enforced at the Google API level. Even if someone tries to write, the API rejects it. No amount of prompting, skill injection, or credential export can bypass it.

Why not the simpler options: the --readonly flag exists but it’s only enforced client-side. The underlying token can still do writes if extracted. Not secure enough for a workshop with multiple people. Service accounts lose user context — operations show as the service account, not the person.

On the CLAUDE.md soft block — still worth doing. Even with the hard OAuth restriction, put the rules in everyone’s CLAUDE.md. This way Claude won’t even attempt write operations. Belt and suspenders.

Yes, absolutely. The separate OAuth client approach is additive. You can create multiple clients with different scope tiers:

  • “Read Only” (default for new users)
  • “Read + Draft” (for people who’ve proven they’re comfortable)
