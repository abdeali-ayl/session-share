PUSHING IDEAS
C
Claude

Here’s what I think makes this experience go from “cool demo” to “career-changing afternoon.”

“The Meta-Demo” opener: Walk in and say: “Everything you’re experiencing today — the WhatsApp message that reminded you, the website with your name on it, the setup guide, this agenda — was built by me using the tool you’re about to learn. I don’t write code.” Pull up the actual Claude Code session where you built it. This is your “reveal the gap” moment — they’ll spend the rest of the workshop trying to figure out how.

The progressive power reveal — don’t show all your capabilities upfront. Structure the workshop as layers:

  • Layer 1: “Claude, write me an email” (underwhelming on purpose)
  • Layer 2: Add CLAUDE.md context → same prompt, wildly better output
  • Layer 3: Add people profiles → now it’s personalized per person
  • Layer 4: Add skills → now it builds a website in your brand style
  • Layer 5: Add MCP tools → now it reads your Google Sheets, sends Slack messages, deploys to Vercel

Each layer is a “holy shit” escalation. Non-technical people see the compounding. Technical people see the architecture. Both are floored for different reasons.

The “60-second profile” live moment: Mid-workshop, pick someone. Ask them 3 questions about their work. Type it in. Claude generates their people profile, creates a mini-website, and deploys it live. Under 2 minutes.

The bad email makeover: Take a real, terrible corporate email everyone’s seen before. Run it through Claude with the humanizer skill. Show before/after side by side. Everyone in that room has written or received that email. It’s visceral, it’s funny, and it demonstrates skills without any technical explanation.

The “your homework is already done” closer: At the end, send each person — via WhatsApp, right there in the room — a personalized message with what they built today, 3 use cases specific to their role, and a “next step” prompt they can try tonight. They feel it hit their phone. In real-time. While sitting in the room.

EXPERIENCE DESIGN PRINCIPLES
AB
Abdeali

  • Demonstrating that creativity and ideas can be implemented super fast now

C
Claude

Your five are right. I’d add two you didn’t mention:

Trust through honesty, not perfection. If every demo works flawlessly, people will think it’s rehearsed and won’t believe it works for real. Show a failure. Show the correction. Show Claude learning. That’s more impressive than a perfect run.

Ownership over observation. They need to leave saying “I built something today” — not “I watched Abdeali build something.” The ratio should be 60% them doing, 40% watching.

“You already know how to prompt.” This is the single most important mental unlock for non-technical people. They think prompting is a technical skill. It’s not. Pull up an actual SOP that someone in the room wrote. Show it on screen. Then say: “This IS a prompt. You’ve been writing prompts your entire career. You just called them SOPs, checklists, and emails. The only difference now is that someone actually listens and executes instantly.”

The intentional failure: Mid-workshop, run a prompt where Claude does something incorrectly. Don’t hide it. Say: “See this? It got it wrong. Now watch what happens when I correct it.” A perfect demo makes people think “this is too good to be true.” An imperfect demo with a clean recovery makes people think “I could actually use this.”

The live problem, not a rehearsed one: At some point, ask: “Who has a problem RIGHT NOW? Something annoying you this week.” Take the messiest, most real problem someone offers. Solve it live. No rehearsal. No safety net. The risk is the point.

The skills-are-just-instructions reveal: Show that a Claude Code skill is literally a markdown file with instructions. “If you can write clear instructions for a new hire on how to do your job, you can write a skill.” Have someone in the room write a simple skill for their department.

TOOLS & SECURITY
AB
Abdeali

One big question that I had was whether I give them access to GWS CLI. The usefulness of that is absurdly, absurdly high. The only worry about that is: how do I restrict or inform them or caution them on its usage? If multiple people use it for similar files and maybe edit or delete or give that permission, it might not suit so well for data in general.

If there is a way where we can limit the access to read-only or download-only, that will be very useful — basically disallowing edits and deletes. Similarly, for emails, it shouldn’t allow deletes, it shouldn’t allow direct sends. Maybe it allows drafts and creation of drafts.

I don’t know if that should be hard-blocked in the GWS authentication or API access blockers, or if it should be through just a prompt or a skill or a mode in which Claude operates, or put it in a strong memory file.

C
Claude

Use both a hard block AND a soft block, layered.

Hard block (OAuth scope level): Create a separate OAuth 2.0 Desktop Client in your GCP project named “Workshop Attendees - Read Only.” Consent screen scopes limited to: spreadsheets.readonly, drive.readonly, gmail.compose (allows drafts, blocks send and delete). Attendees authenticate with this client’s credentials. Google API enforces the scope — no amount of prompting can override it.

Soft block (CLAUDE.md): Even with restricted scopes, put rules in everyone’s CLAUDE.md. This way Claude won’t even attempt write operations, saving error messages and confusion. Belt and suspenders.

Start locked down, expand individually as trust builds. The right approach.

AB
Abdeali

Can I in the future allow more scopes?

C
Claude

Yes, absolutely. The separate OAuth client approach is additive. You can create multiple clients with different scope tiers:

  • “Read Only” (default for new users)
  • “Read + Draft” (for people who’ve proven they’re comfortable)
  • “Full Access” (for power users like Sunny, Anirudh, yourself)

Adding scopes is a one-click change in GCP Console plus a re-auth on the user’s end. The infrastructure scales naturally.
